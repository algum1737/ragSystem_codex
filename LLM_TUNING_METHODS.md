# LLM 튜닝 방법 정리

> 작성일: 2026-05-12  
> 대상: LLM 기반 서비스를 운영하거나 개선하려는 실무자

---

## 튜닝 방법 전체 개요

```
비용·복잡도 낮음 ────────────────────────────── 높음
     │
     ▼
프롬프트 엔지니어링
     │
     ▼
RAG (검색 증강 생성)
     │
     ▼
LoRA / QLoRA (파인튜닝 경량화)
     │
     ▼
Full Fine-tuning
     │
     ▼
RLHF / DPO (정렬 튜닝)
```

---

## 1. 프롬프트 엔지니어링

모델 가중치를 전혀 바꾸지 않고 **입력 설계만으로** 성능을 끌어올리는 방법.

### 주요 기법

| 기법 | 설명 | 예시 |
|---|---|---|
| **Zero-shot** | 예시 없이 지시만 제공 | "이용약관을 요약해줘" |
| **Few-shot** | 입력-출력 예시 2~5개 포함 | Q: "해지하면?" A: "30일 내 환불" 예시 추가 |
| **Chain-of-Thought** | "단계적으로 생각해봐"로 추론 유도 | 복잡한 조항 해석에 효과적 |
| **System prompt** | 역할·규칙·어투를 사전 정의 | "당신은 이용약관 전문 분석가입니다" |

### 적용 조건
- 모델이 이미 해당 도메인 지식을 어느 정도 보유한 경우
- 빠르게 적용하고 결과를 확인해야 할 때
- 별도 학습 인프라가 없을 때

### 한계
- 컨텍스트 창 길이 제한 (긴 문서 처리 어려움)
- 모델이 모르는 정보는 생성 불가 (환각 발생)
- 복잡한 도메인일수록 효과 제한적

---

## 2. RAG (Retrieval-Augmented Generation)

외부 문서를 **검색해서 컨텍스트로 주입**하는 방식. 모델 수정 없음.

### 동작 구조

```
사용자 질문
    │
    ▼
[임베딩 모델] → 벡터 변환
    │
    ▼
[벡터 DB 검색] → 관련 문서 청크 Top-K 추출
    │
    ▼
[LLM] ← 질문 + 검색된 문서를 함께 전달
    │
    ▼
답변 생성
```

### 적용 조건
- 문서가 자주 업데이트되는 경우 (이용약관, 규정, 매뉴얼)
- 출처 추적이 필요한 경우
- 학습 데이터 구축 비용을 피하고 싶을 때

### 품질 향상 포인트
- 임베딩 모델 교체 (예: multilingual-e5-large)
- 청킹 전략 최적화 (크기, 겹침 비율)
- 하이브리드 검색 (BM25 + 벡터 + RRF)
- Cross-Encoder 재순위 (Reranking)

### 한계
- 검색 품질에 답변 품질이 종속됨
- 문서에 없는 내용은 답변 불가
- 실시간 검색 레이턴시 발생

---

## 3. LoRA / QLoRA (경량 파인튜닝)

전체 가중치가 아닌 **일부 레이어만 학습**해 비용을 대폭 줄이는 방식.

### LoRA 원리

```
기존 가중치 행렬 W (고정)
         +
저랭크 행렬 A × B (학습 대상, 파라미터 수 ≪ W)
         =
최종 출력
```

W를 직접 수정하지 않고 작은 행렬 두 개(A, B)만 학습. 학습 파라미터가 전체의 1~5% 수준.

### QLoRA

LoRA + **4bit 양자화** 조합. 48GB RAM 환경에서 gemma3:12b 수준 모델 로컬 파인튜닝 가능.

### 적용 조건
- 특정 도메인 어투·형식 학습 필요 시 (예: 이용약관 분석 특화 답변 형식)
- Q&A 데이터셋 수천 쌍 이상 보유 시
- GPU 또는 대용량 RAM 환경

### 데이터 형식 예시

```json
{
  "instruction": "다음 이용약관 조항에서 해지 관련 내용을 요약해줘.",
  "input": "제10조 (계약 해지) 이용자는 언제든지 서비스 해지를 신청할 수 있으며...",
  "output": "이용자는 언제든지 해지 가능하며, 해지 신청 즉시 처리됩니다."
}
```

### 한계
- 학습 데이터 구축 공수 높음
- 모델 업데이트 시 재학습 필요
- 잘못된 데이터로 학습 시 성능 저하 위험

---

## 4. Full Fine-tuning

모델의 **전체 가중치를 재학습**하는 방식.

### 적용 조건
- 도메인이 매우 특수하여 기존 모델 지식이 거의 없을 때
- 대규모 도메인 데이터(수십만 건 이상) 보유 시
- 전용 GPU 클러스터 보유 시

### 현실적 판단

| 규모 | 권장 여부 |
|---|---|
| 스타트업 / 소규모 팀 | ✗ 비용 대비 효과 낮음 |
| 대기업 / 전문 AI 팀 | △ 필요성이 명확할 때만 |
| 모델 회사 (Anthropic, Google 등) | ✓ 핵심 작업 |

---

## 5. RLHF / DPO (정렬 튜닝)

모델이 **사람이 선호하는 방식**으로 답변하도록 유도하는 방법.

### RLHF (Reinforcement Learning from Human Feedback)

```
1. 모델이 여러 답변 생성
2. 사람이 더 나은 답변 선택
3. 선택 기준으로 보상 모델(Reward Model) 학습
4. 보상 모델 점수로 LLM 강화학습
```

ChatGPT, Claude가 "착하고 안전하게" 동작하는 이유가 이 방식.

### DPO (Direct Preference Optimization)

RLHF에서 보상 모델 학습 단계를 제거한 단순화 버전.  
선호/비선호 답변 쌍만 있으면 바로 학습 가능.

```json
{
  "prompt": "개인정보 삭제 요청은 어떻게 하나요?",
  "chosen": "마이페이지 > 계정 설정 > 개인정보 삭제 요청에서 신청하시면 됩니다.",
  "rejected": "개인정보 삭제는 복잡한 절차가 필요할 수 있습니다."
}
```

### 적용 조건
- 모델이 답변은 하지만 **형식·어투·안전성**을 개선하고 싶을 때
- LoRA 파인튜닝 이후 품질 정교화 단계

---

## 6. 지식 증류 (Knowledge Distillation)

큰 모델(Teacher)의 출력을 학습 데이터로 삼아 **작은 모델(Student)을 훈련**하는 방식.

```
gemma3:27b (Teacher) → 고품질 답변 대량 생성
         │
         ▼
gemma3:4b (Student) → Teacher 답변으로 학습
         │
         ▼
gemma3:4b 수준 모델이 27b에 근접한 성능 달성
```

### 적용 조건
- 서비스 응답 속도·비용 개선이 필요할 때
- 경량 모델을 특정 도메인에 특화시킬 때

---

## 방법별 비교 요약

| 방법 | 모델 수정 | 데이터 필요량 | 비용 | 지식 업데이트 |
|---|---|---|---|---|
| 프롬프트 엔지니어링 | ✗ | 없음 | 거의 없음 | 즉시 |
| RAG | ✗ | 원본 문서 | 낮음 | 문서 교체만 |
| LoRA / QLoRA | ✓ (일부) | 수천~수만 건 | 중간 | 재학습 필요 |
| Full Fine-tuning | ✓ (전체) | 수십만 건 이상 | 매우 높음 | 재학습 필요 |
| RLHF / DPO | ✓ (정렬) | 선호도 쌍 수천 건 | 높음 | 재학습 필요 |
| 지식 증류 | ✓ (학생 모델) | Teacher 출력 대량 | 중간 | 재학습 필요 |

## 이용약관 조회 시스템 기준 추천 순서

```
현재 ──▶ 1. 프롬프트 엔지니어링 (PROMPT_TEMPLATE 개선)
          2. RAG 품질 향상 (임베딩·재순위·청킹 최적화)  ← 현재 진행 중
          3. QLoRA 파인튜닝 (이용약관 Q&A 데이터 충분히 쌓인 후)
          4. DPO (답변 형식·어투 정교화)
```

RAG만 잘 구성해도 이용약관처럼 **문서 기반 도메인**에서는 파인튜닝 없이 높은 품질을 달성할 수 있습니다.
