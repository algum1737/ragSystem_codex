RAG LLM 응답 지연 구간 분석

운영 RAG API의 api.query trace를 기준으로 모델별 전체 응답 시간, DB 조회, 검색 파이프라인, LLM 생성 시간을 비교했다. 결론적으로 실제 Vector DB 조회는 병목이 아니며, 대부분의 지연은 LLM 생성 구간에서 발생한다.

작성일: 2026-06-08 대상: ragSystem 운영 API trace: /opt/ragSystem_codex/logs/rag_traces.jsonl 단위: 초

핵심 요약

주 병목은 LLM 생성 검색이 warm 상태일 때도 LLM 생성만 7초에서 200초 이상 소요된다.
Vector DB 조회는 매우 짧음 실제 Chroma vector search는 대체로 0.005초에서 0.06초 수준이다.
cold load는 보조 병목 embedding/reranker 초기 로드 시 검색 전체가 9초 이상으로 튈 수 있다.

구간 정의

전체 시간
API가 질문을 받아 검색, 프롬프트 구성, LLM 생성, 응답 반환까지 완료하는 총 시간.
검색 전체
embedding + vector_search + bm25_search + rerank를 합친 retrieval 구간.
Embedding
질문을 임베딩 벡터로 변환하는 시간. cold load 시 가장 크게 증가한다.
Vector DB 조회
Chroma에서 유사 벡터를 검색하는 순수 DB 조회 시간.
BM25
키워드 기반 검색 시간.
Rerank
검색 후보를 Cross-Encoder로 재정렬하는 시간.
LLM 생성
Ollama 모델이 최종 답변 텍스트를 생성하는 시간.

결론

운영 응답 지연의 핵심 원인은 DB 조회가 아니라 LLM 생성이다. gemma4:26b는 LLM 생성만 200초 이상 걸리는 사례가 있어 운영 기본값으로 부적합하다. 현재 운영 기본값인 gemma3:12b + top_k=5는 품질을 크게 유지하면서 지연을 7초에서 22초 수준으로 낮춘 조합이다.

다음 개선은 기본 token cap보다 concise-answer prompt 실험이 더 적합하다. token cap은 이미 낮은 값에서 답변 절단과 품질 회귀가 확인됐고, 높은 값은 latency 개선 효과가 작았다.
