# Quality Score

## Purpose

품질 상태를 문서, 코드, 검증 루프 기준으로 평가한다.

## Dimensions

- 문서 최신성
- 테스트 우선 구현 준수 수준
- 작업 단위 리뷰와 완료 전 검증 수준
- 검증 자동화 수준
- 아키텍처 일관성
- 운영 관측 가능성
- 기술부채 관리 수준

## Scoring Rule

- 각 항목을 `0-3`으로 점수화한다.
- 총점과 함께 근거 문서를 남긴다.
- 점수 하락 원인은 후속 작업과 연결한다.

## Current Baseline

- 문서 최신성: `2`
  - 하네스 진입 문서는 정리됐지만 세부 설계 문서는 일부 원본 프로젝트 표현을 유지한다.
- 검증 자동화 수준: `1`
  - 문서 검증 스크립트는 있으나 런타임 smoke test, lint, typecheck는 아직 연결되지 않았다.
- 아키텍처 일관성: `2`
  - 코드 경계는 비교적 분명하지만 제품 명칭과 문서 구조가 방금 전환된 상태다.
- 운영 관측 가능성: `1`
  - 서버 로그는 있으나 구조화된 운영 관측 루프는 미정이다.
- 기술부채 관리 수준: `2`
  - `.paul`의 이력은 문서로 흡수 중이며 후속 정리 항목이 명시돼 있다.

## Superpowers-Aligned Checks

- 기능 추가와 버그 수정은 가능한 한 실패하는 테스트 또는 재현 eval을 먼저 남긴다.
- 테스트나 eval이 즉시 통과했다면 기존 동작을 검증한 것인지, 새 요구사항을 검증한 것인지 확인한다.
- task 완료 전에는 계획 요구사항 충족 여부와 코드 품질을 별도로 검토한다.
- 완료 선언 전에는 실제 실행한 검증 명령, eval 지표, trace/latency 결과를 `Validation Result`에 남긴다.
- 반복되는 리뷰 지적은 문서 규칙, lint, test, eval, CI 후보로 승격한다.
